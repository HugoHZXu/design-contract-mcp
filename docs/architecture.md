# Architecture Notes

[English](architecture.md) | [简体中文](architecture.zh-CN.md)

## Overview

Design Contract MCP is a **contract-first Figma-to-Code MCP server**. When a component library is connected to Figma via Code Connect, this MCP fetches design frame data through the Figma MCP, loads the AI contract artifact generated from the component library, resolves component mappings, builds a generation context pack, and validates generated React code against the contract.

The MCP logic is largely complete, but the repository currently uses local mock fixtures shaped to mirror real Figma MCP output and Code Connect mappings, because it has not yet been wired up against a live Figma environment for integration testing. The fixtures allow the complete MCP pipeline — contract resolution, context building, validation — to run end-to-end without a live Figma connection. When connecting to real Figma, only the data source layer needs to be replaced; the core MCP logic stays the same.

The repository is organized around a single data flow:

```text
Figma frame data (mock fixtures currently, live Figma MCP when integrated)
  → normalized design fixture
  → Code Connect component mappings
  → @hugo-ui/mui AI contract artifact
  → generation context pack
  → AI-generated React (generated by the calling client)
  → validation report
```

## Repository Relationship

[`hugo-ui`](https://github.com/HugoHZXu/hugo-ui) owns the design-system source and publishes versioned `@hugo-ui/mui` AI contract artifacts through GitHub Releases.

Design Contract MCP consumes those artifacts in two ways:

- `vendor/hugo-ui/mui-ai-contract/` provides a committed fallback snapshot so the project works out of the box.
- `.cache/hugo-ui/mui-ai-contract/<version>/` stores synced release artifacts for local development or deployed runs.

## Directory Roles

Here's what each key directory contains and how it fits into the pipeline:

| Directory | Purpose |
|-----------|---------|
| `vendor/hugo-ui/mui-ai-contract/` | Fallback contract snapshot with component contracts, token contracts, schemas, metadata, and `provenance.json` for source traceability |
| `.cache/hugo-ui/mui-ai-contract/<version>/` | Populated by `npm run contract:sync:hugo-ui` or startup sync; runtime tools resolve contracts from here first, falling back to vendor |
| `fixtures/figma/mcp/` | Raw captured Figma MCP tool results |
| `fixtures/figma/` | Normalized design fixtures produced by [normalize-figma-fixture.ts](file:///Users/xuhaoze/code-demo/figma-contract-mcp-demo/scripts/normalize-figma-fixture.ts); this is the direct input consumed by MCP tools |
| `code-connect/manifest.json` | Maps design node IDs to `@hugo-ui/mui` component names and contract file paths |
| `code-connect/mock/` | Local Code Connect template-shape examples for mapped components |
| `contracts/patterns/` | Page-level pattern contracts. The current example uses [modal-form.pattern.json](file:///Users/xuhaoze/code-demo/figma-contract-mcp-demo/contracts/patterns/modal-form.pattern.json) for the edit profile modal flow |
| `mcp-server/` | Core MCP server code: stdio/HTTP/HTTPS entrypoints, local CLI, contract adapter, context tools, and validator |
| `generated/` | Output artifacts: generation context pack, sample React candidates, and validation reports |

## Context Flow

Let's walk through the data pipeline step by step, using the built-in Edit Profile Modal example:

1. **Raw capture** — [edit-profile-modal.mcp-context.json](file:///Users/xuhaoze/code-demo/figma-contract-mcp-demo/fixtures/figma/mcp/edit-profile-modal.mcp-context.json) stores the captured Figma MCP result: sparse XML metadata, React-like design context with `CodeConnectSnippet` markers, a `get_code_connect_map` result, and variable definitions.

2. **Normalization** — Running `npm run figma:normalize` invokes [normalize-figma-fixture.ts](file:///Users/xuhaoze/code-demo/figma-contract-mcp-demo/scripts/normalize-figma-fixture.ts), which converts the raw capture into [edit-profile-modal.fixture.json](file:///Users/xuhaoze/code-demo/figma-contract-mcp-demo/fixtures/figma/edit-profile-modal.fixture.json), preserving node IDs, component IDs, typed properties, selected layout data, Code Connect snippets, text values, and component metadata.

3. **Mapping resolution** — [manifest.json](file:///Users/xuhaoze/code-demo/figma-contract-mcp-demo/code-connect/manifest.json) resolves selected design nodes to `@hugo-ui/mui` components and their contract paths.

4. **Contract resolution** — [contract-store.ts](file:///Users/xuhaoze/code-demo/figma-contract-mcp-demo/mcp-server/src/contract-store.ts) resolves the requested contract selector against local cache entries and the vendor fallback.

5. **Contract adaptation** — [hugo-ui-mui.ts](file:///Users/xuhaoze/code-demo/figma-contract-mcp-demo/mcp-server/src/contract-adapters/hugo-ui-mui.ts) adapts the published `hugo-ui` contract shape into the validator's internal format while preserving raw contract data for traceability.

6. **Pattern rules** — [modal-form.pattern.json](file:///Users/xuhaoze/code-demo/figma-contract-mcp-demo/contracts/patterns/modal-form.pattern.json) adds page-level structure and generation guidance for modal forms.

7. **Context assembly** — [tools.ts](file:///Users/xuhaoze/code-demo/figma-contract-mcp-demo/mcp-server/src/tools.ts) combines the normalized fixture, mapping metadata, resolved contracts, token policy, and pattern contract into a single generation context pack.

8. **Context pack output** — Running `npm run context:pack` produces [edit-profile-modal.context-pack.json](file:///Users/xuhaoze/code-demo/figma-contract-mcp-demo/generated/edit-profile-modal.context-pack.json), which records the resolved chain, contract version, contract source, and expected component usage.

9. **Validation** — [validator.ts](file:///Users/xuhaoze/code-demo/figma-contract-mcp-demo/mcp-server/src/validator.ts) validates generated React against imports, props, forbidden props, raw colors, and expected mapped component coverage.

## Contract Version Resolution

The design-system release process decides when to publish a new AI contract artifact. A new artifact is called for whenever generated code shape, supported props, token policy, validation rules, or documented AI usage guidance changes.

The MCP side supports these selectors:

| Selector | Behavior |
|----------|----------|
| `vendor` | Uses the committed fallback snapshot |
| `latest` | Uses the newest locally available cached or vendored contract |
| `installed` | Reads the local `@hugo-ui/mui` package version and selects the newest local contract with version ≤ that package version |
| Semver (e.g. `1.0.5`) | Selects the newest local contract with version ≤ the specified target |

When you run `npm run contract:sync:hugo-ui`, it applies the same `latest`/`installed`/semver resolution logic against remote `mui-ai-contract-v*` releases, verifies the downloaded checksum, unpacks the artifact into `.cache/`, and checks provenance metadata before runtime tools will use it.

## MCP Runtime

`npm run mcp:server`, `npm run mcp:http`, and `npm run mcp:https` all share the same server factory and tool registration. The entrypoints differ only in transport and where TLS termination happens:

- **stdio** — for local MCP clients and debugging
- **Streamable HTTP** — for HTTP-based clients and gateways
- **Node HTTPS** — for deployments where this process terminates TLS directly

The server exposes a set of context and validation tools (see the [README](../README.md#running-the-mcp-server) for the full tool list and descriptions).

For internal deployments, place the HTTP or HTTPS entrypoint behind the platform component that owns authentication, authorization, request logging, and secret management. Set `MCP_AUTH_MODE=external` to record that upstream authentication mode in health checks and logs. `MCP_ALLOWED_HOSTS` and `MCP_ALLOWED_ORIGINS` provide hooks for host and browser-origin filtering.

> 💡 See the main [README](../README.md#running-the-mcp-server) for detailed configuration options, environment variables, certificate setup, and the complete tool list.

## Validation

Validation is the final gate before generated React is presented as usable output. The validator checks:

- Imports come from the expected component packages
- JSX props are all listed in the adapted component contract
- Forbidden props are not used
- All expected mapped components from `expectedComponentUsage` are present in the generated JSX
- No raw color literals such as `#FF0000`, `rgb(...)`, or `hsl(...)` are hardcoded

Validation failures are returned as structured report data. Local commands and MCP clients can display these reports directly to users or downstream tooling.

## Integration Points for Real Design Flows

The current fixture-based path maps cleanly to a real internal workflow. The main integration points are:

1. Replace the fixture reader with a design context provider backed by an internal Figma MCP service or design-ingestion service
2. Publish the mapping artifact from the design-system or design-platform workflow (rather than maintaining it manually)
3. Sync contract artifacts automatically during setup, deployment, or process startup
4. Keep code generation in the calling AI application (this server intentionally does not generate code)
5. Feed generated code back into `validate_generated_code` before review or handoff to developers

## Contract Adapter

The [hugo-ui-mui.ts](file:///Users/xuhaoze/code-demo/figma-contract-mcp-demo/mcp-server/src/contract-adapters/hugo-ui-mui.ts) adapter converts the published `hugo-ui` contract format into the validator's internal representation, while always retaining the raw contract and provenance data. This ensures every validation decision can be traced back to the exact published artifact version.
