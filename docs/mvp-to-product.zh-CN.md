# 从模拟数据到生产环境

[English](mvp-to-product.md) | [简体中文](mvp-to-product.zh-CN.md)

本文档说明如何将这个 MCP 从当前状态（使用与真实 Figma MCP 和 Code Connect 数据格式一致的本地模拟 fixture 运行）逐步演进为连接真实 Figma 的生产级 Figma-to-Code 工作流。核心思路保持不变：我们始终围绕**设计上下文 → 组件映射 → 版本化 AI 合约 → 生成上下文 → 代码生成指导 → 校验反馈**这条链路来构建。

---

## 1. 设计数据接入

### 当前状态

仓库在 [`fixtures/figma/mcp/`](../fixtures/figma/mcp/) 下提供了一份提前捕获好的 Figma MCP 数据样例。运行 `npm run figma:normalize` 时，[`normalize-figma-fixture.ts`](../scripts/normalize-figma-fixture.ts) 会把原始数据转换成标准化的 [`edit-profile-modal.fixture.json`](../fixtures/figma/edit-profile-modal.fixture.json)，这就是 MCP 工具消费的输入格式。

### 生产环境路径

在真实工作流中，同样的标准化数据格式应该由内部设计数据源来提供，而不是静态 fixture。可以接入的来源包括：

- 内部 Figma MCP 服务
- 受控的设计数据采集服务
- 选定 frame 或节点的缓存快照
- 稳定的 frame/节点标识符体系
- 设计快照的新鲜度元数据（用于提示用户设计是否有更新）
- 当文件、frame 或节点无法访问时，给用户清晰的错误提示

这样 MCP 上下文工具只需要消费已授权的设计数据，不需要直接持有 Figma 工作区的访问权限。

---

## 2. 组件映射管理

### 当前状态

[`code-connect/manifest.json`](../code-connect/manifest.json) 定义了设计节点 ID 到 `@hugo-ui/mui` 组件的映射关系，以及对应的合约文件路径。[`code-connect/mock/`](../code-connect/mock/) 目录下提供了 `Modal`、`Input`、`Button` 三个组件的 Code Connect 模板示例。

### 生产环境路径

生产环境的组件映射应该由设计系统团队或设计平台团队来维护。根据团队的技术栈，可以选择官方 Code Connect、内部组件注册表，或者自动生成的映射产物。

建议补充以下能力：

- 随设计系统一起发布的**版本化映射产物**，保证映射与组件版本对应
- 校验映射条目中引用的组件确实存在于 AI 合约清单中
- 过期映射和缺失映射的自动检测
- 支持多个包或组件命名空间
- 为组件迁移提供废弃标记和别名元数据

## 3. AI 合约管理

### 当前状态

仓库在 [`vendor/hugo-ui/mui-ai-contract/`](../vendor/hugo-ui/mui-ai-contract/) 下提交了一份兜底的合约快照。运行时也可以从 `.cache/hugo-ui/mui-ai-contract/<version>/` 读取通过同步命令下载的合约产物。

同步命令 (`npm run contract:sync:hugo-ui`) 会从 GitHub Releases 下载合约包，校验 checksum，解压到缓存目录，并检查 `provenance.json` 溯源信息。运行时 MCP 工具优先从本地缓存读取，找不到时回退到 vendor 快照。

支持的版本选择器：
- `vendor` — 使用提交的兜底快照
- `latest` — 使用最新版本
- `installed` — 匹配本地 `@hugo-ui/mui` 包版本
- 具体版本号（如 `1.0.2`）

### 生产环境路径

在内部产品中，设计系统的发布流程应该把**不可变**的 AI 合约包发布到内部制品仓库，比如 S3、GCS、OSS、内部 npm registry，或者其他经过审批的制品存储服务。

推荐的同步模型：

```text
设计系统发布
  → 发布不可变的合约包和校验文件
  → 更新制品索引
部署流水线或初始化步骤
  → 选择需要支持的合约版本
  → 下载并校验制品
  → 解压到本地只读缓存
MCP 运行时
  → 从本地缓存解析合约
```

建议补充：

- 维护一个制品索引，记录哪些版本是受支持的、哪些是最新的、哪些已废弃
- 支持按明确版本、版本范围或受支持版本集合来同步
- 本地缓存的清理策略（避免旧版本无限堆积）
- 除了 GitHub Releases，增加对象存储作为制品来源
- 部署时检查必需的合约版本是否就绪
- 明确发布新 AI 合约的触发条件：代码生成规则、支持的 props、token 策略、校验规则发生变化时就应该发布新版本

## 4. MCP 服务部署

### 当前状态

仓库提供了三种 MCP 服务入口：

- `npm run mcp:server` — 本地 stdio 模式，用于调试
- `npm run mcp:http` — Streamable HTTP 模式
- `npm run mcp:https` — 需要由 Node 直接处理 TLS 时使用

HTTP 和 HTTPS 模式共用同一套 MCP 服务工厂、请求处理器、合约解析器、健康检查、Host 过滤、浏览器来源过滤，以及结构化的 stderr 日志。`MCP_AUTH_MODE=external` 表示认证由上游平台组件处理。

### 生产环境路径

内部使用时，MCP 服务通常部署在企业网关、内部平台、白名单云服务或私有网络边界之后。由平台层负责 TLS 终止、认证、授权、请求日志、密钥管理、日志留存和策略集成。

当部署环境确定后，建议补充：

- 对应环境的 Dockerfile 和部署示例
- 在启动或初始化阶段自动同步合约
- 就绪检查（readiness check），确保必需的合约版本已加载
- 对接平台的日志和监控系统
- 明确配置 Host 和浏览器来源白名单
- 针对部署端点的冒烟测试

## 5. 校验与用户反馈

### 当前状态

校验器会根据已解析的组件合约和上下文包中的预期组件清单来检查生成的 React 代码，报告以下问题：

- 导入包不匹配
- 使用了合约之外的 props
- 使用了被禁用的 props
- 缺少应有的组件覆盖
- 硬编码的原始颜色值

本地校验命令和 MCP 校验响应都会完整展示从设计到映射到合约到代码到校验结果的全链路信息。

### 生产环境路径

在保持"合约优先"核心链路不变的前提下，真实产品可以增加更丰富的校验和反馈信号：

- 对生成代码运行 TypeScript 编译和 lint 检查
- 针对支持的模式做无障碍（a11y）检查
- 对高价值流程做视觉回归测试
- 当平台策略要求时，集中留存校验记录
- 把每一条校验失败关联到对应的合约、映射或模式规则，给用户清晰可理解的解释
- 对于缺少映射或映射过期的设计 frame，建立评审队列让人工处理

## 推荐实施顺序

从当前使用模拟数据的状态出发，落地到生产环境 Figma-to-Code 工作流的一条实际可行的路径：

1. **保留模拟流水线**：继续使用本地 fixture 作为 MCP 逻辑的可复现测试基准
2. **抽象数据源接口**：在 fixture 读取器旁边增加设计上下文提供者接口，方便后续替换为真实 Figma MCP 客户端
3. **接入真实 Code Connect 映射**：将静态 manifest 替换为设计系统发布的真实 Code Connect 映射数据
4. **自动化合约同步**：把合约同步移到安装、部署或进程启动阶段自动完成
5. **部署到内部平台**：把 HTTP 或 HTTPS MCP 入口部署到内部网关之后
6. **补充下游校验**：在生成代码进入评审的环节增加 TypeScript、无障碍和视觉检查

无论怎么扩展，MCP 的核心工作流始终是：解析设计上下文 → 映射到组件合约 → 构建生成上下文包 → 通过合约指导代码生成 → 校验结果。这条链路的清晰度是整个项目可维护性的基础。
